# 千瓜数据看板与洞察日报设计

日期：2026-07-02

## 背景

当前项目是一个轻量 Python 服务：用户上传千瓜 Excel 文件后，系统解析入 SQLite，调用 MiniMax 完成 4 层分析，再生成一份 HTML 看板。现有实现可用，但上传、入库、LLM 分析、HTML 生成和页面服务都集中在 `qiangua_upload_server.py`，页面也把原始数据、趋势聚类、爆款笔记和选题建议混在一张长页面里。

目标是参考 `https://aihot.virxact.com/daily` 的日报式信息架构，把“数据看板”和“洞察分析”分开，形成每天可上传、可分析、可回看、可持续迭代的网页服务。

## 参考站点观察

`AI HOT Daily` 的核心设计不是复杂视觉效果，而是清晰的阅读结构：

- 左侧是主导航和日报历史，可快速切换日期。
- 主内容是一期日报，顶部有期号、日期、故事数等元信息。
- 内容按类别分组，每组有序号、标题、英文辅助标签和条目数量。
- 每条内容是标题、来源、摘要和详情链接，适合深度阅读。
- 页面是偏编辑部风格的日报，而不是操作型数据后台。

这个模式适合用在“洞察分析”页，不适合直接套到“数据看板”页。数据看板应更像运营分析台，强调筛选、排序、指标和明细表。

## 产品结构

采用一个网页服务、两个主工作区：

1. 数据看板：`/dashboard`
2. 洞察日报：`/insights`

保留上传入口：`/upload` 或首页快速上传区。

### 顶层导航

- 上传数据
- 数据看板
- 洞察日报
- 历史记录
- 设置

### 日期维度

所有页面都以 `analysis_date` 为核心维度。页面顶部统一提供日期选择器：

- 最新一期
- 最近 7 天
- 指定日期
- 历史归档

## 数据看板页

定位：回答“今天数据表现如何、哪些内容/热词/话题值得看、可筛选明细在哪里”。

页面建议：

- 顶部指标条：笔记数、热词数、话题数、总互动、爆款笔记数、低粉爆款数。
- 趋势概览：热词增长排行、话题增长排行、互动排行。
- 内容明细：笔记表格，支持按互动、点赞、收藏、评论、粉丝数、内容形式、来源类型筛选。
- 热词明细：热词、热度、笔记增长、商业笔记增长、分类。
- 话题明细：话题、浏览增长、互动增长、笔记增长。
- 原始文件记录：上传文件、识别类型、入库行数、状态、时间。

交互重点：

- 以表格、排序、过滤和搜索为主。
- 每个图表都能下钻到原始笔记或热词。
- 不把 LLM 长文分析塞进看板主视图，只显示必要的摘要标签和跳转。

视觉方向：

- 浅色、密集、操作型。
- 使用窄边框、紧凑表格、固定工具栏。
- 图表和表格比大卡片更重要。

## 洞察日报页

定位：回答“今天应该怎么理解这些趋势，Intel 可以做什么内容”。

参考 `AI HOT Daily`，采用日报阅读器结构：

- 左侧：日报历史列表，按月分组，显示日期和当日主标题。
- 主内容顶部：期号、日期、数据规模、生成时间。
- 开篇摘要：今日总判断，3 到 5 条重点结论。
- 主题分组：
  - 趋势聚类
  - 深度机会
  - 爆款笔记拆解
  - 内容趋势
  - 选题推荐
- 每个分组内部使用文章式条目，而不是指标卡。
- 每条洞察包含：标题、证据、判断、Intel 角度、可执行建议、引用笔记或话题。

建议页面路由：

- `/insights`：最新一期日报。
- `/insights/2026-07-02`：指定日期日报。
- `/insights/archive`：全部归档。

日报内容顺序：

1. 今日一句话判断
2. 重点机会 Top 3
3. 趋势聚类
4. 爆款笔记拆解
5. 选题建议
6. 数据依据和原始链接

## 上传与处理流程

保留当前上传体验，但把处理状态做成更明确的任务流：

1. 上传 Excel 或压缩包。
2. 文件识别与解压。
3. 入库并记录上传批次。
4. 生成基础数据快照。
5. 运行 L1 趋势聚类。
6. 运行 L2 深度机会分析。
7. 运行 L3 爆款笔记分析。
8. 运行 L4 选题推荐。
9. 生成看板数据和洞察日报数据。

上传完成后进入任务详情页，而不是只给一个 HTML 链接：

- 当前状态
- 每一步耗时
- 错误信息
- 可打开数据看板
- 可打开洞察日报

## 数据边界

当前数据库表可保留：

- `xhs_notes`
- `xhs_hotwords`
- `xhs_topics`
- `xhs_analysis`
- `upload_log`

建议新增或调整：

- `analysis_runs`：一次分析任务的状态、日期、开始结束时间、模型、错误。
- `upload_batches`：一次上传批次，关联多个文件。
- `analysis_artifacts`：结构化分析产物，按类型保存 JSON。

`xhs_analysis` 可以继续使用，但命名建议标准化：

- `l1_clusters`
- `l2_opportunities`
- `l3_note_patterns`
- `l4_recommendations`
- `raw_notes_snapshot`
- `dashboard_summary`
- `daily_report`

## 服务边界

短期不建议上来就拆成前后端双项目。最稳妥的迁移路径是：

1. 保持单个 Python 服务。
2. 从 `qiangua_upload_server.py` 拆出模块：
   - `db.py`
   - `ingest.py`
   - `analysis.py`
   - `reports.py`
   - `server.py`
3. 先用服务端模板渲染页面。
4. 后续如果页面复杂，再迁移到 FastAPI + React 或 Next.js。

建议接口：

- `POST /api/uploads`
- `GET /api/runs`
- `GET /api/runs/{run_id}`
- `GET /api/dashboard?date=YYYY-MM-DD`
- `GET /api/insights?date=YYYY-MM-DD`
- `GET /api/history`

页面路由：

- `GET /`
- `GET /upload`
- `GET /dashboard`
- `GET /insights`
- `GET /insights/{date}`
- `GET /history`

## 错误处理

必须明确处理：

- 文件类型不支持。
- 文件名无法识别千瓜数据类型。
- Excel 列名不匹配。
- 重复上传。
- LLM 没有 API key。
- LLM 返回非 JSON。
- 某一层分析失败但基础看板仍可展示。

原则：数据看板可以在没有 LLM 的情况下展示；洞察日报需要标记哪些部分缺失。

## 测试与验收

最低验收标准：

- 上传页可上传多个 Excel 或压缩包。
- 数据看板和洞察日报是两个独立页面。
- 同一天上传后，两个页面都能通过日期访问。
- 没有 LLM key 时，看板仍可用，并显示洞察不可用原因。
- 历史日期可切换。
- 页面移动端不重叠，桌面端有清晰左侧历史导航。

建议测试：

- Python 编译检查。
- 单元测试：文件类型识别、数字解析、JSON 解析。
- 数据库测试：重复上传、日期查询、分析产物保存。
- 页面快照测试：上传页、看板页、洞察页。

## 推荐实施顺序

1. 先修文档与启动说明，明确 Python 依赖和运行命令。
2. 抽出数据访问与分析产物保存逻辑。
3. 新增 `/dashboard` 和 `/insights` 两套页面，不再只生成一个静态 HTML。
4. 将当前 `dashboard_generator.py` 改为洞察日报的结构化渲染器。
5. 为数据看板新增表格和筛选视图。
6. 增加历史归档和任务状态页。

## 设计决策

选择“一个服务、两个工作区”，而不是完全拆成两个服务。原因是当前项目体量小，数据源和分析任务相同，过早拆服务会增加部署和调试成本。真正需要分开的是用户心智和页面结构：数据看板偏操作，洞察日报偏阅读。
